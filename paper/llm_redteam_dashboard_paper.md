# An Open and Deployable Dashboard for LLM Red-Teaming and Qualitative Evaluation

## Abstract

This technical report describes a Streamlit dashboard for safe public demonstration of qualitative LLM red-team evaluation. The implemented repository provides deterministic demo benchmark artifacts, Docker/Hugging Face Spaces deployment configuration, guarded runtime settings, and reproducibility documentation. The reported benchmark results are demo artifacts generated from repository YAML cases, not live model measurements.

## Introduction

Public LLM evaluation demos need to show meaningful qualitative behavior without exposing API keys or allowing strangers to spend model quota. This project addresses that deployment constraint with a safe demo mode and sample benchmark report layer.

## System Overview

The main dashboard entrypoint is `app.py`. Runtime settings are loaded from environment variables in `src/runtime_config.py`. Demo benchmark artifacts are generated by `python -m src.generate_demo_benchmark`. The guarded CLI entrypoint is `python -m src.run_redteam --config evals/config.yaml`.

## Architecture

| Component | Path | Purpose |
|---|---|---|
| Streamlit dashboard | `app.py` | Public UI and settings visibility |
| Benchmark loader/generator | `src/redteam_benchmark.py` | Deterministic demo report generation |
| Runtime settings | `src/runtime_config.py` | Safe environment-driven mode control |
| Guarded runner | `src/runner.py` | Demo execution and live-run gate |
| Eval config | `evals/config.yaml` | Default benchmark/report configuration |
| Cases | `benchmarks/qualitative_redteam_cases.yaml` | Safe qualitative test cases |

## Deployment Model

The Hugging Face Space uses Docker with `app_port: 8501`. Public variables should set `DEMO_MODE=true` and `ALLOW_LIVE_RUNS=false`. The app does not require permanent disk for the public demo; reports can be regenerated deterministically.

## Benchmark Design

The demo benchmark covers prompt injection resistance, unsafe instruction refusal, hallucination risk, ambiguous request handling, judge failure handling, and config/path handling. Cases are safe and abstract, avoiding harmful operational detail.

## Qualitative Evaluation

The current demo artifact contains six cases.

| Fail | 0 |

These values come from `reports/demo_benchmark/demo_results.json`, generated from `benchmarks/qualitative_redteam_cases.yaml`.

## Error Handling and Reproducibility

The demo benchmark is deterministic and does not call external APIs. Missing report directories are created during generation. Live execution fails closed unless explicitly enabled and keyed.

## Public Demo Mode and Safety Considerations

Demo mode disables live model calls and does not require `OPENAI_API_KEY`. The dashboard can indicate whether a key exists without displaying it. Session-provided keys are not persisted.

## Limitations

The live provider path currently targets OpenAI Chat Completions with a small default case limit. Demo results are illustrative and must not be presented as live model performance. Hugging Face free storage is ephemeral.

## Conclusion

The repository is suitable for a portfolio-style public demo of safe LLM red-team evaluation workflows. It prioritizes deployability, transparency, and clear separation between demo artifacts and live evaluation capability.

## Reproducibility Checklist

1. Install: `python -m pip install -r requirements.txt`
2. Generate demo benchmark: `python -m src.generate_demo_benchmark`
3. Run tests: `python -m pytest tests -q`
4. Start app: `streamlit run app.py`
5. Deploy HF Space with Docker, port 8501, `DEMO_MODE=true`, `ALLOW_LIVE_RUNS=false`
